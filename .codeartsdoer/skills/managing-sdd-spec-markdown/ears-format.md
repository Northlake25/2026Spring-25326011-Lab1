# EARS 格式说明

## 什么是 EARS 格式？

EARS (Easy Approach to Requirements Syntax) 是一种简化的需求编写格式，旨在使需求更加清晰、可测试和可验证。

## EARS 格式结构

每个需求应遵循以下结构：

### 1. 需求ID
- 格式：`REQ-XXX-NNN`
- XXX：需求领域缩写（如 MAT-比赛, PLR-球员, EVT-事件）
- NNN：序号（001, 002, 003...）

### 2. 需求描述
使用以下标准句式：

#### 2.1 通用型需求
```
系统应[功能描述]
```
示例：系统应支持创建新的足球比赛记录。

#### 2.2 事件驱动型需求
```
当[触发条件]时，系统应[响应行为]
```
示例：当用户点击"开始比赛"按钮时，系统应记录比赛开始时间并进入比赛进行状态。

#### 2.3 状态驱动型需求
```
当[状态条件]时，系统应[功能描述]
```
示例：当比赛处于进行中状态时，系统应允许记录换人、进球、红黄牌等事件。

#### 2.4 可选功能型需求
```
如果[可选条件]，系统应[功能描述]
```
示例：如果进球有助攻球员，系统应记录助攻球员信息。

#### 2.5 复杂条件型需求
```
当[条件A]且[条件B]时，系统应[功能描述]
```
示例：当比赛未结束且球员未被替换下场时，系统应允许该球员参与比赛事件。

## 验收标准

每个需求应包含明确的验收标准，使用以下格式：

```
验收标准：
- [可测试条件1]
- [可测试条件2]
- [可测试条件3]
```

示例：
```
验收标准：
- 用户能够输入比赛名称、日期、时间和地点
- 系统生成唯一的比赛ID
- 比赛初始状态为"未开始"
- 首发名单和替补名单初始为空
```

## 需求分类

### 功能需求 (FR)
描述系统应提供的功能。

### 非功能需求 (NFR)
描述系统的质量属性：
- 性能需求
- 安全需求
- 可用性需求
- 兼容性需求

### 约束需求 (CR)
描述系统的技术或业务约束。

## 需求优先级

- P0：核心功能，必须实现
- P1：重要功能，应该实现
- P2：增强功能，可以实现
- P3：未来考虑

## 编写原则

1. **原子性**：每个需求只描述一个功能点
2. **可测试性**：需求必须有明确的验收标准
3. **无歧义性**：使用清晰、明确的语言
4. **完整性**：包含所有必要信息
5. **一致性**：不与其他需求冲突
