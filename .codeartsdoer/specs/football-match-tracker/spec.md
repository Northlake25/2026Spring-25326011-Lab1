# 需求规格文档 - 足球比赛记录器

## 文档信息

- **项目名称**：足球比赛记录器 (Football Match Tracker)
- **特性名称**：football-match-tracker
- **版本号**：v1.0
- **创建日期**：2024-04-14
- **最后更新**：2024-04-14

## 1. 概述

### 1.1 项目背景
本项目旨在开发一款运行在 HarmonyOS 平台上的足球比赛记录应用程序。该应用允许用户按时间顺序记录一场足球比赛的完整进程，包括首发名单、替补名单、换人事件、进球事件（含助攻）、红黄牌事件等核心比赛信息。通过系统化的记录方式，帮助用户准确、完整地保存比赛数据，并支持后续的数据查看和统计分析。

### 1.2 功能范围
本次需求涵盖以下核心功能：
- **比赛管理**：创建、查看、编辑和删除比赛记录
- **球队与球员管理**：管理参赛球队信息和球员名单
- **首发阵容管理**：设置和查看首发阵容及替补名单
- **比赛事件记录**：按时间顺序记录换人、进球、助攻、红黄牌等事件
- **比赛进程跟踪**：实时显示比分、比赛时间、事件时间线
- **数据统计**：自动统计比赛数据（进球数、红黄牌数等）

### 1.3 排除范围
以下功能不在本次需求范围内：
- 视频回放和视频分析功能
- 实时数据同步到云端服务器
- 多用户协作记录功能
- 比赛数据导出为PDF或Excel格式
- 球员详细技术统计（如跑动距离、传球成功率等）
- 赛事积分榜和排名系统
- 社交分享功能

## 2. 用户角色

### 2.1 角色定义
| 角色 | 描述 | 权限 |
|------|------|------|
| 比赛记录员 | 负责记录比赛进程的用户 | 创建比赛、编辑比赛信息、记录比赛事件、查看统计数据 |
| 观众 | 查看比赛信息的用户 | 查看比赛列表、查看比赛详情、查看事件时间线 |

## 3. 功能需求

### 3.1 比赛管理

#### REQ-MAT-001: 创建新比赛
**优先级**：P0
**描述**：系统应支持用户创建新的足球比赛记录。

**验收标准**：
- 用户能够输入比赛基本信息：主队名称、客队名称、比赛场地、比赛日期、比赛时间、赛事名称、裁判
- 系统为每场比赛生成唯一的比赛ID
- 比赛创建后初始状态为"未开始"（scheduled）
- 比赛初始比分为主队0-0客队
- 比赛事件列表初始为空
- 创建成功后自动跳转到比赛详情页面

**业务规则**：
- 主队名称和客队名称不能为空
- 比赛日期和时间必须有效
- 主队和客队不能是同一支球队

#### REQ-MAT-002: 查看比赛列表
**优先级**：P0
**描述**：系统应支持用户查看所有比赛记录的列表。

**验收标准**：
- 列表显示每场比赛的赛事名称、对阵双方、比分、日期时间、场地
- 列表按比赛日期倒序排列（最新的比赛在前）
- 每场比赛卡片提供"查看详情"和"删除"操作按钮
- 如果没有比赛记录，显示空状态提示

#### REQ-MAT-003: 查看比赛详情
**优先级**：P0
**描述**：系统应支持用户查看单场比赛的详细信息。

**验收标准**：
- 显示比赛基本信息：赛事名称、主客队、比分、场地、日期时间、裁判、比赛状态
- 显示当前比赛时间（如果比赛进行中）
- 显示比赛事件时间线（按时间顺序）
- 提供"记录事件"、"首发阵容"、"统计数据"操作按钮
- 比分根据进球事件自动更新

#### REQ-MAT-004: 删除比赛
**优先级**：P1
**描述**：当用户点击删除按钮时，系统应提示确认删除，确认后删除比赛记录。

**验收标准**：
- 点击删除按钮时弹出确认对话框
- 确认对话框显示"确定要删除这场比赛吗？"
- 用户确认后删除比赛记录
- 删除成功后从列表中移除该比赛
- 删除操作不可撤销

#### REQ-MAT-005: 开始比赛
**优先级**：P0
**描述**：当用户点击"开始比赛"按钮时，系统应记录比赛开始时间并将比赛状态改为"进行中"。

**验收标准**：
- 比赛状态从"未开始"变为"进行中"（in_progress）
- 系统记录比赛开始时间戳
- 开始计时，实时显示比赛进行时间
- 比赛时间从0'开始计时

#### REQ-MAT-006: 结束比赛
**优先级**：P0
**描述**：当用户点击"结束比赛"按钮时，系统应记录比赛结束时间并将比赛状态改为"已结束"。

**验收标准**：
- 比赛状态从"进行中"变为"已结束"（finished）
- 系统记录比赛结束时间戳
- 停止计时
- 保留所有已记录的比赛事件

### 3.2 球队与球员管理

#### REQ-TEM-001: 球队信息管理
**优先级**：P0
**描述**：系统应支持管理球队的基本信息。

**验收标准**：
- 每个球队包含：球队ID、球队名称、球队颜色、球队Logo（可选）
- 球队名称不能为空
- 球队颜色用于UI显示区分不同球队

#### REQ-PLR-001: 球员信息管理
**优先级**：P0
**描述**：系统应支持管理球员的基本信息。

**验收标准**：
- 每个球员包含：球员ID、所属球队ID、姓名、球衣号码、位置、是否队长
- 球员姓名和球衣号码不能为空
- 球员位置包括：守门员、后卫、中场、前锋
- 同一球队的球衣号码不能重复

#### REQ-PLR-002: 添加球员到球队
**优先级**：P1
**描述**：系统应支持为球队添加新球员。

**验收标准**：
- 用户能够输入球员姓名、球衣号码、位置、是否队长
- 系统为球员生成唯一ID
- 球员自动关联到当前球队
- 球衣号码在同一球队内唯一

### 3.3 首发阵容管理

#### REQ-LIN-001: 设置首发阵容
**优先级**：P0
**描述**：系统应支持用户为每支球队设置首发阵容。

**验收标准**：
- 用户能够从球队球员名单中选择首发球员
- 首发阵容最多包含11名球员
- 首发阵容至少包含1名守门员
- 未被选为首发的球员自动归入替补名单
- 首发阵容设置后生成首发阵容事件

**业务规则**：
- 标准足球比赛首发阵容为11人
- 必须包含且仅包含1名守门员

#### REQ-LIN-002: 查看首发阵容
**优先级**：P0
**描述**：系统应支持用户查看比赛的首发阵容和替补名单。

**验收标准**：
- 分别显示主队和客队的首发阵容
- 显示每位首发球员的姓名、号码、位置
- 标识队长球员
- 显示替补球员名单

#### REQ-LIN-003: 修改首发阵容
**优先级**：P1
**描述**：当比赛未开始时，系统应允许修改首发阵容。

**验收标准**：
- 比赛状态为"未开始"时可以修改首发阵容
- 修改后更新首发阵容事件
- 比赛开始后不允许修改首发阵容

### 3.4 比赛事件记录

#### REQ-EVT-001: 记录换人事件
**优先级**：P0
**描述**：当比赛进行中时，系统应支持记录换人事件。

**验收标准**：
- 用户选择球队、被换下球员、换上球员
- 系统记录换人时间（比赛时间）
- 生成换人事件并添加到事件列表
- 被换下球员标记为已下场
- 换上球员标记为已上场
- 事件描述格式："[球队] [换上球员号码] [换上球员姓名] 换下 [被换下球员号码] [被换下球员姓名]"

**业务规则**：
- 被换下球员必须是当前场上球员
- 换上球员必须是替补球员且未上场
- 同一球员不能被重复换上或换下

#### REQ-EVT-002: 记录进球事件
**优先级**：P0
**描述**：当比赛进行中时，系统应支持记录进球事件。

**验收标准**：
- 用户选择进球球队、进球球员
- 系统记录进球时间（比赛时间）
- 用户可选择是否为点球进球
- 用户可选择是否为乌龙球
- 生成进球事件并添加到事件列表
- 自动更新比分
- 事件描述格式："[球队] [球员号码] [球员姓名] 进球"

**业务规则**：
- 进球球员必须是当前场上球员
- 乌龙球计入对方球队得分
- 点球进球需要特殊标记

#### REQ-EVT-003: 记录助攻
**优先级**：P0
**描述**：如果进球有助攻球员，系统应支持记录助攻信息。

**验收标准**：
- 在记录进球时可以选择助攻球员
- 助攻球员信息与进球事件关联
- 事件描述更新为："[球队] [球员号码] [球员姓名] 进球 (助攻: [助攻球员号码] [助攻球员姓名])"

**业务规则**：
- 助攻球员必须是当前场上球员
- 助攻球员与进球球员不能是同一人
- 助攻球员与进球球员应为同一球队（乌龙球除外）

#### REQ-EVT-004: 记录黄牌
**优先级**：P0
**描述**：当比赛进行中时，系统应支持记录黄牌事件。

**验收标准**：
- 用户选择球队、犯规球员
- 系统记录黄牌时间（比赛时间）
- 生成黄牌事件并添加到事件列表
- 事件描述格式："[球队] [球员号码] [球员姓名] 黄牌"

**业务规则**：
- 同一球员累计两张黄牌自动转换为红牌
- 记录第二张黄牌时标记为"两黄变一红"

#### REQ-EVT-005: 记录红牌
**优先级**：P0
**描述**：当比赛进行中时，系统应支持记录红牌事件。

**验收标准**：
- 用户选择球队、犯规球员
- 系统记录红牌时间（比赛时间）
- 用户可选择是否为"两黄变一红"
- 生成红牌事件并添加到事件列表
- 被罚下球员标记为已下场
- 事件描述格式："[球队] [球员号码] [球员姓名] 红牌"

**业务规则**：
- 被罚红牌的球员必须立即下场
- 球队不能换人补充被罚下的球员

#### REQ-EVT-006: 事件时间线显示
**优先级**：P0
**描述**：系统应按时间顺序显示所有比赛事件。

**验收标准**：
- 事件按发生时间顺序排列（从早到晚）
- 每个事件显示：事件时间、事件图标、事件类型、事件描述
- 不同类型事件使用不同图标和颜色区分
- 支持滚动查看所有事件

#### REQ-EVT-007: 中场休息
**优先级**：P1
**描述**：当上半场结束时，系统应支持记录中场休息事件。

**验收标准**：
- 用户点击"中场休息"按钮
- 比赛状态变为"中场休息"（halftime）
- 生成中场休息事件
- 比赛时间显示为"45'"
- 暂停计时

#### REQ-EVT-008: 开始下半场
**优先级**：P1
**描述**：当用户点击"开始下半场"按钮时，系统应继续比赛。

**验收标准**：
- 比赛状态从"中场休息"变为"进行中"
- 比赛时间从"45'"开始继续计时
- 允许继续记录比赛事件

### 3.5 数据统计

#### REQ-STA-001: 比赛统计
**优先级**：P1
**描述**：系统应自动统计比赛数据。

**验收标准**：
- 统计主队和客队的进球数、黄牌数、红牌数、换人次数
- 统计数据实时更新
- 以卡片形式展示统计数据
- 支持对比显示两队数据

#### REQ-STA-002: 球员统计
**优先级**：P1
**描述**：系统应统计每位球员在比赛中的数据。

**验收标准**：
- 统计每位球员的进球数、助攻数、黄牌数、红牌数
- 显示球员姓名、号码和统计数据
- 按进球数排序显示球员列表

## 4. 非功能需求

### 4.1 性能需求
- 应用启动时间不超过3秒
- 比赛列表加载时间不超过1秒
- 事件记录响应时间不超过500毫秒
- 支持至少100场比赛记录的存储
- 界面滚动流畅，帧率不低于60fps

### 4.2 安全需求
- 比赛数据仅存储在用户设备本地
- 不收集和传输用户个人信息
- 数据持久化存储，应用关闭后数据不丢失

### 4.3 可用性需求
- 界面简洁直观，操作步骤最少化
- 关键操作提供确认提示（如删除比赛）
- 错误操作提供友好的错误提示
- 支持触摸操作，按钮和可点击区域大小适中
- 重要信息使用醒目颜色和图标标识

### 4.4 兼容性需求
- 支持 HarmonyOS 系统版本 4.0 及以上
- 适配不同屏幕尺寸（手机、平板）
- 支持横屏和竖屏显示
- 适配深色模式和浅色模式

## 5. 约束条件

### 5.1 技术约束
- 开发平台：HarmonyOS
- 开发语言：ArkTS (TypeScript)
- UI框架：ArkUI 声明式开发范式
- 数据存储：本地内存存储（可扩展为Preferences或RDB）
- 不使用第三方云服务

### 5.2 业务约束
- 遵循标准足球比赛规则
- 比赛时长默认为90分钟（上下半场各45分钟）
- 换人次数限制为每队最多5次（可配置）
- 球员名单最多可包含30名球员

## 6. 数据需求

### 6.1 数据实体

#### 比赛实体 (FootballMatch)
- id: string - 比赛唯一标识
- homeTeam: Team - 主队信息
- awayTeam: Team - 客队信息
- venue: string - 比赛场地
- date: string - 比赛日期
- time: string - 比赛时间
- competition: string - 赛事名称
- referee: string - 裁判
- homeScore: number - 主队得分
- awayScore: number - 客队得分
- status: MatchStatus - 比赛状态
- startTime: number - 开始时间戳
- endTime: number - 结束时间戳
- events: MatchEvent[] - 事件列表

#### 球队实体 (Team)
- id: string - 球队唯一标识
- name: string - 球队名称
- color: string - 球队颜色
- logo: string - 球队Logo路径

#### 球员实体 (Player)
- id: string - 球员唯一标识
- teamId: string - 所属球队ID
- name: string - 球员姓名
- number: number - 球衣号码
- position: PlayerPosition - 球员位置
- isCaptain: boolean - 是否队长

#### 比赛事件实体 (MatchEvent)
- id: string - 事件唯一标识
- matchId: string - 所属比赛ID
- type: MatchEventType - 事件类型
- timestamp: number - 事件时间戳
- matchTime: string - 比赛时间
- description: string - 事件描述

### 6.2 数据关系
- 一个比赛包含两个球队（主队和客队）
- 一个球队包含多个球员
- 一个比赛包含多个比赛事件
- 一个事件关联一个比赛
- 进球事件可选关联助攻球员
- 换人事件关联两个球员（换上和换下）

### 6.3 数据验证规则
- 球队名称：非空，长度1-50字符
- 球员姓名：非空，长度1-30字符
- 球衣号码：1-99之间的整数
- 比赛日期：有效的日期格式 YYYY-MM-DD
- 比赛时间：有效的时间格式 HH:MM
- 比赛时间：格式为 "MM'" 或 "MM+XX'"

## 8. 验收测试场景

### 8.1 比赛管理

#### 场景1.1: 创建新比赛
**前置条件**：用户在主页或比赛列表页
**操作步骤**：
1. 点击"新建比赛"按钮
2. 输入主队名称"曼联"
3. 输入客队名称"利物浦"
4. 输入比赛场地"老特拉福德"
5. 输入比赛日期"2024-04-20"

## 9. 术语表

| 术语 | 定义 |
|------|------|
| 首发阵容 | 比赛开始时上场的11名球员 |
| 替补名单 | 未在首发阵容中，但可在比赛中被换上场的球员 |
| 换人 | 比赛中用替补球员替换场上球员的行为 |
| 助攻 | 进球前最后一次传球的球员，直接导致进球 |
| 乌龙球 | 球员将球踢入自家球门，得分计入对方球队 |
| 点球 | 因犯规在禁区内判罚的罚球 |
| 黄牌 | 警告性处罚，累计两张黄牌将变为红牌 |
| 红牌 | 罚下性处罚，球员必须立即离场 |
| 两黄变一红 | 同一球员累计两张黄牌，自动转为红牌 |
| 比赛时间 | 从比赛开始计算的经过时间，格式如"45'"表示45分钟 |
| 伤停补时 | 上下半场结束时增加的补时时间，格式如"45+2'" |
